## 2. Actual Training Flow

Current flow:

1. Hand image goes into `data/images/`.
2. Annotation goes into `data/palmistry_llava.json`.
3. Launcher script passes paths and LoRA settings into `src/train/train_sft.py`.
4. `train_sft.py` loads `Qwen3VLForConditionalGeneration`, applies LoRA, builds the supervised dataset, and writes adapters/checkpoints into `output/`.

The training wrapper is intentionally thin. It does not replace the upstream trainer. It only standardizes:

- model path
- data path
- image folder
- output dir
- LoRA hyperparameters
- multi-GPU / DeepSpeed knobs

## 3. Current Dataset Shape

The main dataset is LLaVA-style:

- one `image`
- one user turn with `<image>`
- one assistant turn

Important observation:

- The assistant target in the current palmistry dataset is mostly a stringified JSON report generated by GPT-5.
- That means the LoRA is learning image-conditioned structured palmistry interpretation first.
- The final "natural Chinese report" behavior is mostly injected by inference prompt design, not by a separate polished reporting SFT stage.

This is workable, but it also means:

- if you want more stable long-form natural reports, add a second-stage dataset where assistant outputs are already natural prose
- if you want better structured extraction, keep the current JSON-style labels

## 4. Inference Flow

The new canonical inference flow is:

1. Load base Qwen3-VL model
2. Attach LoRA adapter if provided
3. Build a multimodal chat message with the actual image object
4. Use palmistry prompt templates to control report tone and structure
5. Decode the model output and optionally continue the conversation with follow-up QA

Core files:

- [src/palmistry/pipeline.py](../src/palmistry/pipeline.py)
- [src/palmistry/prompts.py](../src/palmistry/prompts.py)

## 5. Bug Fixed In The New Canonical Entry Points

The older local inference scripts in the project root used `{"type": "input_image"}` as a placeholder, but they did not attach the real uploaded image into the message content passed to `process_vision_info(...)`.

`qwen_vl_utils` expects content like:

- `{"type": "image", "image": <PIL image or file path>}`

The new pipeline fixes that, so the model is actually conditioned on the hand image.
